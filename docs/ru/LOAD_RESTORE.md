# Загрузка сессии (`--load`)

**`niri-session-manage --load-last`** — загрузка из файла, заданного в конфиге полем **`[session].graceful_shutdown_name`** (по умолчанию файл **`last`** в [каталоге сессий](CONFIG.md#секция-session-каталог-сессий)); по смыслу это **`--load`** с заранее известным путём (тот же снимок, что сохраняет **`--graceful-shutdown`**). Тайминги, **`[[launch]]`** и таблица флагов ниже применяются так же, как к **`--load`**.

---

`niri-session-manage --load [файл]` (без аргумента — **`session.json`** в [каталоге сессий](CONFIG.md#секция-session-каталог-сессий)):

1. Читает JSON (см. [SESSION_FORMAT.md](SESSION_FORMAT.md)).
2. Сортирует окна по `(output, workspace_idx, column, tile)`.
3. Склеивает подряд идущие **тайловые** окна с одним `(output, workspace_idx, column)` в группы (**колонка с несколькими тайлами**).
4. Для каждой группы (или одиночного окна) по очереди:
   - фокусирует нужный монитор (`FocusMonitor`) и стол (`FocusWorkspace`);
   - если в группе **несколько** окон: перед каждым spawn — `FocusColumn` с сохранённым **1-based** индексом колонки; при **включённом ожидании** (по умолчанию) после spawn опрашивается `Request::Windows`, пока не появится новый id окна или не истечёт **`spawn_deadline`**; после **второго и следующих** запусков затем вызывается **`ConsumeWindowIntoColumn`** (окно справа втягивается в эту колонку);
   - выбирает **argv для запуска** (`[[launch]]` при необходимости — см. [CONFIG.md](CONFIG.md));
   - запускает процесс через `std::process::Command` (не через IPC `Spawn`);
   - паузы — из `[load]` / CLI (`ipc_settle_ms`, `spawn_start_delay_ms`).

**По умолчанию** после каждого успешного spawn загрузка **ждёт** появления нового окна в списке niri (до **`spawn_deadline`** мс), затем переходит к следующему — так соблюдается порядок старта. **`--no-await`** или **`[load].no_await = true`** отключают ожидание («запустил и забыл»; для колонок с несколькими тайлами **`ConsumeWindowIntoColumn`** может сработать слишком рано).

Одиночные окна и плавающие режимы без `ConsumeWindowIntoColumn`. Порядок в JSON и паузы задают, на каком столе окажется фокус перед каждым стартом.

Перед каждым `spawn` проверяется, **нет ли уже подходящего окна** на целевом столе: для тайлов — тот же выход/индекс стола, позиция `(column, tile)` и `app_id`; для плавающих — `app_id` и заголовок (подстрока). Если такое окно есть, шаг **пропускается** (дубликат не запускается). Чтобы всё равно запускать процессы, задайте **`--open-forcefully`** или **`[load].open_forcefully = true`** в конфиге.

**Ошибки по отдельным окнам** (нет `[[launch]]`, сбой `spawn`, пустая команда, ошибка IPC при фокусе) **не прерывают** загрузку: такое окно пропускается, в stderr пишется `окно пропущено: …`, остальные окна обрабатываются. Если была хотя бы одна ошибка, в конце выводится итог и **код выхода 1**; при полном успехе — **0**.

## Параметры `--load` (CLI, env, конфиг)

Все значения — в **миллисекундах**.

**Приоритет:** аргумент CLI → переменная окружения → секция **`[load]`** в `niri-session.conf` → встроенное значение по умолчанию.

| Флаг | Env | Дефолт (без конфига) | Назначение |
|------|-----|----------------------|------------|
| `--ipc-settle-ms` | `NIRI_SESSION_IPC_SETTLE_MS` | `80` | Пауза после IPC фокуса (монитор/стол) и **после каждого успешного spawn** перед следующим шагом. |
| `--spawn-start-delay-ms` | `NIRI_SESSION_SPAWN_START_DELAY_MS` | `0` | Дополнительная пауза после spawn перед следующим окном (разгрузка CPU/диска при массовом старте). |
| `--no-await` | — | ожидание **вкл.** | Не ждать появления нового окна после spawn; следующий шаг сразу после пауз `ipc_settle` / `spawn_start_delay`. |
| `--spawn-deadline` | `NIRI_SESSION_SPAWN_DEADLINE_MS` | `10000` | Максимум миллисекунд ожидания нового id окна в niri после spawn (пока ожидание не отключено). |
| `--no-notify-on-spawn-failure` | `NIRI_SESSION_NOTIFY_ON_SPAWN_FAILURE` (`true`/`false`/`0`/`1`) | уведомления **вкл.** | Не вызывать `notify-send` при ошибках запуска (нет правила `[[launch]]`, не удалось выполнить `spawn`, пустая команда). |
| `--open-forcefully` | — | выкл. | Не пропускать шаг, если подходящее окно уже есть; всегда выполнять `spawn` (см. также `[load].open_forcefully` в [CONFIG.md](CONFIG.md)). |
| `--config` | — | — | Путь к TOML (`[load]` + `[[launch]]`). Без флага: `~/.config/niri-session/niri-session.conf`, если есть. |
| `-d` / `--debug` | — | выкл. | Подробный журнал в **stderr**: `NIRI_SOCKET`, каждый IPC-запрос/ответ, окна при `--save`, разбор команды и `spawn` при `--load`, паузы. |
| `--load-last` | — | — | Загрузка из **`[session].graceful_shutdown_name`**; см. [CONFIG.md](CONFIG.md) (подраздел **`--graceful-shutdown` и `--load-last`**). |

Уведомления: при сбое подготовки команды или `spawn` вызывается **`notify-send`** (пакет `libnotify`), если не отключено.

Пример:

```sh
niri-session-manage --load ~/session.json --ipc-settle-ms 120 --spawn-start-delay-ms 200
```

Постоянные настройки удобно держать в `[load]` в `niri-session.conf` (см. [CONFIG.md](CONFIG.md)).

## Конфиг `[[launch]]` (обязателен для xwayland-satellite и т.п.)

Если сохранённая `command` **непереносимая** (встроенная проверка: `-listenfd`; для мостов вроде **xwayland-satellite** задайте **`resolve`** в правиле), при `--load` без подходящего правила в TOML будет ошибка. См. `resolve` и порядок сопоставления в [CONFIG.md](CONFIG.md). Пример: [niri-session.conf.example](../niri-session.conf.example).

Альтернатива: вручную заменить `command` у окна в JSON — но тогда теряется автоматическое «как в /proc» при следующем `--save`.

## PWA, Flatpak

Предпочтительно прописать `[[launch]]` с `command = ["flatpak", "run", …]` или `app_id` + `title_contains` для нужного окна. См. [CONFIG.md](CONFIG.md).

## X11 и xwayland-satellite

Окна X11 в niri идут через **xwayland-satellite**; в JSON часто сохраняется `argv` вида `xwayland-satellite … -listenfd …`. Для загрузки добавьте в конфиг правила с **`resolve = "xwayland-satellite"`** (и при необходимости `resolve = "-listenfd"`) и `app_id` — см. [TROUBLESHOOTING.md](TROUBLESHOOTING.md) и [CONFIG.md](CONFIG.md).

## Ограничения

- **Несколько окон в одной колонке** (`column` + разные `tile` в JSON): при `--load` окна запускаются по возрастанию `tile`; после каждого следующего окна вызывается IPC **`ConsumeWindowIntoColumn`** (новое окно справа притягивается в колонку с индексом **`column`**). Перед каждым spawn выполняется **`FocusColumn`** с этим индексом. Порядок тайлов внутри стека и режим колонки (табы и т.д.) могут **не совпасть** с сохранённым снимком; при медленном появлении окон срабатывает дедлайн опроса — увеличьте **`spawn_deadline`** (или **`ipc_settle_ms`** как интервал опроса).
- **Плавающие окна** и **точная геометрия** тайлов: как раньше, без переноса позиций после старта.
- **PID / fork:** сопоставление окна с процессом после spawn не используется; новое окно ищется по **разнице** множеств id из `Request::Windows`.
